# <a name="컴퓨터 역사 " href="#컴퓨터 역사 ">컴퓨터 역사 </a>

**컴퓨터의 역사에 관한 메모를 해 둔 곳입니다.**

---

튜링 기계를 실제로 구현하자 이후의 컴퓨터 과학은 두 갈래로 갈라졌다. 하나는 알고리즘으로써 소프트웨어(튜링기계)가 어떻게 일하게 할 것인가 에 대한 연구였고, 다른 하나는 프로그래밍 언어로써 소프트웨어(튜링기계) 를 어떻게 표현할 것인가 에 대한 연구였다. 

# <a name="알고리즘 " href="#알고리즘 ">알고리즘 </a>

- 소프트웨어는 튜링기계이다. 즉 프로그램은 튜링기계이다. 컴퓨터는 그 튜링기계를 읽고 흉내내는 보편만능 튜링기계이다. 

- 알고리즘이란 소프트웨어(튜링기계) 로 만드는 문제 푸는 방법이다. 

## <a name="복잡도 " href="#복잡도 ">복잡도 </a>

- 복잡도complexity 란 알고리즘을 실행할 때 드는 비용이다.

- 주어진 문제를 기계적으로 푸기 위하여 (1) 알고리즘(문제풀이법) 을 찾고 (2) 그 해법이 맞는지 확인한 후 (3) 복잡도가 감당할만하면 (4) 알고리즘대로 소프트웨어를 만든다.

- 그러나 멈춤 문제처럼 컴퓨터가 절대 못푸는 문제도 많다. 또 풀 수 있다고 해도 그 비용이 너무 커서 현실적으로 못푸는 문제도 많다.

- 복잡도에서 관심을 두어야할 것은 입력의 크기에 따라서 비용이 어떻게 증가하느냐 이다. 

  > p101~p134

# <a name="소프트웨어" href="#소프트웨어">소프트웨어</a>

- 소프트웨어(튜링기계) 를 컴퓨터(보편만능 튜링기계) 의 입력으로 주기 위하여 그것을 표현할 언어가 필요하다. 이것이 프로그래밍 언어이다.

- 튜링기계의 규칙표 방식은 너무 원시적이고 수학적 증명을 위한 최소한의 골격만을 갖고 있다.

  - 따라서 근본적으로 모든 소프트웨어가 튜링기계의 규칙표로 정의될 수 있으나 현실은 끝없이 정교하고 복잡한 일을 체계적이고 효율적이고 빠르게 만들 수 있는 프로그래밍 언어가 필요하다. 

## <a name="기계와 사람의 간격" href="#기계와 사람의 간격">기계와 사람의 간격</a>

- 알고리즘을 찾으면 그것을 소프트웨어(튜링기계) 로 만든다. 그러면 컴퓨터(보편만능 튜링기계)가 그것을 실행한다. 

  - 이때 소프트웨어는 언어라는 수단으로 사상되어야만 한다. 

- 그렇다면 사람에게 편한 언어와 컴퓨터에게 쉬운 언어가 일치하는가?

  - 튜링기계는 소프트웨어를 표현하기에 충분한 언어이다.

  - 하지만 튜링기계는 사람이 쓰기에 너무 원시적이다.

    - 심지어 덧셈하는 튜링기계 작동규칙표를 만드는 일도 보통일이 아니다. 
  
  - 좀 더 개선되었긴 했으나 폰 노이만이 구현한 컴퓨터가 사용하는 언어인 기계어도 매한가지이다.

    - 기계어는 튜링기계보다 편하지만 그 언어도 사람이 쓰기엔 너무 원시적이다. 

- 이 사람과 기계의 간격을 해결하기 위하여 상위 언어와 번역compilation 이 사용된다.

  - 컴퓨터가 직접 이해하는 언어는 기계어(튜링기계) 이다. 

  - 그런데 이 언어를 사람이 사용하기 편하게 상위 언어로 사상시킬 수 있고, 이 상위 언어는 다시 기계어로 번역될 수 있다. 

    - 가령 
    
      **Coq** &rarr; **ML**, **Python**, **Scala** &rarr; **Java** &rarr; **C** &rarr; **Dalvik** 바이트 코드 &rarr; **x86-64** 기계어 
      
      순으로 상위 언어라고 할 수 있다. 

### <a name="튜링 완전" href="#튜링 완전">튜링 완전</a>

- 그런데 이러한 프로그래밍 언어마다 표현력의 차이가 있는가? 그러니까 어떤 프로그래밍 언어는 표현하지 못하는 튜링기계를 다른 프로그래밍 언어는 표현할 수 있는가? 

  - 표현력이 완전한 프로그래밍 언어를 튜링-완전turing-complete 하다고 한다.

  - 튜링-완전한 프로그래밍 언어는 모든 소프트웨어(튜링기계)를 표현할 수 있다.

  - 튜링완전한 언어는 튜링 기계로 할 수 있는 모든 계산을 다 할 수 있으므로 범용 언어general-purpose language 라고 한다. 

- 상위 언어를 기계어로 번역할 때 상위 언어가 튜링-완전하다면 안심이 된다. 

  - 왜냐하면 사람이 쓰기 편하면서 온전하게 기계어로 번역될 수 있기 때문이다.

  - 튜링-불완전 언어를 사용하면 번역이 막히는 경우가 발생할 수 있다. 

## <a name="자동 번역 " href="#자동 번역 ">자동 번역 </a>

- 자연어는 자동 번역이 힘들지만 프로그래밍 언어는 항상 자동 번역이 가능하다. 

  - 왜냐하면 자연어는 한 단어가 여러 의미를 지니지만 프로그래밍 언어는 모든 문장 부품의 의미가 유일하기 때문이다. 

  - 프로그래밍 언어를 자동 번역해주는 프로그램을 번역기compiler 라고 한다. 

  > 이것으로 초코드체계의 가능성을 엿볼 수 있다. 

  > 튜링-완전한 프로그래밍 언어는 항상 서로 번역 가능하다는 말이구나.

- 자동번역은 다음 두 원리를 사용한다. 

  1. 부품에서 전체로(조립식compositional) : 전체 번역 결과가 부분 번역 결과로 만들어진다는 뜻이다.

      - 이때 부품 조립 방법은 유한개이다. 불 논리에서 명제를 "그리고", "또는", "아닌" 으로 조립했듯이 프로그램도 부품의 조립이며 그 조립 규칙은 유한하다. 

      - 이건 마치 공리와 유한 개의 추론규칙으로 정리를 연역해내는 과정과 같다. 

  2. 불변성질invariant 유지하기 : 부분들의 번역 결과가 항상 유지하는 성질이 있다는 것이다. 

  - 이 두 원리가 서로 의존하며 자동번역을 해낸다. 부분을 번역한 결과는 불변하는 성질이 유지되고, 그 불변성 덕분에 전체 번역 결과를 속편하게 조립할 수 있게 된다. 

  - 예시 

    출발어source language 와 도착어target language 를 정하여 예를 들어보자. 출발어는 정수식이다. 정수 하나 $n$ 도 정수식이 될 수 있고 $-$ 를 붙힌 $-n$ 도 정수식이 될 수 있고 두 정수 $n, m$ 를 $+$ 로 묶은 $n+m$ 도 정수식이 될 수 있다. 

    도착어는 $3$ 가지 명령 `push n`, `add`, `minus` 로 구성된다. 이 명령이 일렬로 늘어서있는 것이 문장이 된다. 이 명령들은 스택 구조의 메모리의 데이터를 다루며 그 의미는 다음과 같다. `push n` 은 스택에 `n` 을 쌓는다. `add` 는 스택 맨 위에서 두개의 데이터를 꺼내서 더하고 더한 결과를 다시 넣는다. `minus` 는 스택 맨 위의 데이터의 부호를 바꾸고 다시 넣는다. 

    출발어에서 도착어로 번역하는 규칙을 정할 수 있다. 

    정수식의 번역 결과가 유지하는 불변성질은 이러하다. 

    > p149~152 이해 안됨 

- 이렇게 튜링-완전한 프로그래밍 언어 사이에는 자동 번역이 가능하다.

  - 이 자동 번역 덕분에 상위 언어를 사용하여 소프트웨어를 조립하면 자동 번역기가 기계어로 번역해주고 컴퓨터가 이것을 실행하게 된다. 

## <a name="실행 " href="#실행 ">실행 </a>

- 컴퓨터는 소프트웨어를 자동 실행하는데 이것을 해석실행interpretation 이라고 한다. 

  - 왜냐하면 컴퓨터(보편만능 튜링기계) 메모리에 표현된 튜링기계(소프트웨어) 기호들의 의미를 해석해서 그 튜링기계의 작동을 그대로 흉내내기 때문이다. 

  - 즉 컴퓨터는 번역사슬 맨 밑바닥 언어, 즉 기계어의 실행기이다. 

  - 상위 언어로 짠 소프트웨어가 기계어로 번역되면 컴퓨터가 실행할 수 있다.

- 실행기가 반드시 하드웨어일 필요는 없다. 언어의 실행기를 소프트웨어로도 만들 수 있다.

  > 파이썬, 자바 처럼 

  - 이 실행기는 소프트웨어를 번역 없이 해석실행한다. 단, 이 실행기는 소프트웨어 이므로 기계어로 번역되어 컴퓨터 메모리에 실려있어야 한다. 

## <a name="또 다른 방식의 프로그래밍 언어" href="#또 다른 방식의 프로그래밍 언어">또 다른 방식의 프로그래밍 언어</a>

- 기계적으로 계산한다는 것, 즉 컴퓨터로 계산한다는 것의 정의를 어떻게 하느냐에 따라 다른 방식의 언어가 만들어진다. 

- 튜링은 기계적으로 계산 가능한 것을 정의하며 테이프를 읽고 규칙표를 따라 테이프의 문자를 읽고 쓰는 튜링기계를 제시했다. 

- 알론조 처치는 기계적으로 계산 가능한 것을 람다 계산법이라는 개념으로 정의했다. 

  - 이 람다 계산법과 튜링 기계의 표현력은 똑같다. 즉 모든 튜링기계는 람다 계산법으로 계산할 수 있고 반대도 그렇다. 

- 튜링이 정의한 기계적 계산인 튜링기계로부터 상위 언어로 올라온 언어들은 기계를 염두해둔 언어가 되었고, 

  처치가 정의한 기계적 계산인 람다 계산법으로부터 상위 언어로 올라온 언어들은 기계는 없고 값들이 함수를 통과하며 새 값으로 만들어지는 과정이 주를 이룬다. 

  - 이처럼 기계 중심 프로그래밍 언어에는 **C**, **Java**, **C++**, **C#**, **Javascript** 등이 있고 

    함수 중심 프로그래밍 언어에는 **Haskell**, **ML**, **F#**, **Clojure**, **Lisp** 등이 있으며 

    둘 다 지원하는 언어는 **ML**, **F#**, **C++11**, **Python**, **Scala**, **Clojure**, **Lisp** 등이 있다. 
  
- 이 두 패러다임의 언어 중 무엇이 더 좋냐고 논쟁하는 건 무의미하다. 

  - 왜냐하면 각각의 언어마다 적절하게 사용되는 상황이 따로 있기 때문이다.
  
  - 육상 선수가 더 낫냐, 수영 선수가 더 낫냐고 논쟁하는 격이다. 

- 이 두 패러다임의 언어가 프로그래밍 언어의 두 가지 시초이지만, 튜링도 처치도 프로그래밍 언어의 원시 모델을 만들려는 것이 아니라 단지 기계적인 자동계산에 대한 각자의 정의를 내린 것일 뿐이었다. 

  - 이 둘 모두 **"기계적인 계산, 자연적 대상을 기술로 대체하여 자동화하는 것이란 이것이다!"** 라고 정의한 것이고, 
  
    이는 곧 **"이런 것이 컴퓨터 프로그램(튜링기계) 를 표현하는 것의 전부이다!"** 라고 선언한 것이고,
    
    결국 프로그래밍 언어를 소개한 것이다.

### <a name="튜링 기계의 패러다임 " href="#튜링 기계의 패러다임 ">튜링 기계의 패러다임 </a>

- 튜링기계의 프로그래밍 언어 패러다임에서 프로그램이란 기계에게 전달하는 일련의 명령어이다.

  - 즉, 프로그램(튜링기계)은 기계의 현재 상태에서 다음 상태를 뭐로 할지, 메모리에 뭘 쓸지 명령하는 것에 불과하다.

  ### > 완전 엄청나게 단순하게 말하면 "이것이면 저것을 하라" 이고 좀 더 구체적으로 말하면 "이것일 때 현재 상태를 다음 상태로 바꾸고, 테이프에 이것을 써라" 가 되는 것이구나. 그리고 이것을 더욱 구체적으로 말하면 레지스터, 메모리, 기계어, 가상 메모리 시스템, 캐시메모리 등의 설명이 들어가야 하는 것이고.

  - 이 패러다임에서는 프로그램의 일련의 명령어대로만 실행하면 되는 것이다. 즉, 언어란 기계에 전달하는 명령어들을 표현하는 언어인 것이다. 

  - 그리고 그 명령어들이란 기계의 상태(레지스터, 메모리, 드라이브)를 변경시키는 명령에 불과하다. 

### <a name="람다 계산법의 패러다임 " href="#람다 계산법의 패러다임 ">람다 계산법의 패러다임 </a>

- 하지만 람다 계산법 패러다임에서 프로그램은 기계의 상태를 변화시키는 일련의 명령어가 아니다.

  - 프로그램은 단지 함수와 함수 사이를 통과하는 데이터들이다. 

  - 마치 단지 수학 $1+2$ 나 $f(n) = n ^{2}$ 에 대하여 $\sum_{n=1}^{10}f(n)$ 을 계산하는 것처럼 수식의 계산에 불과하다.

  - 따라서 이 관점에서는 기계와 기계의 상태를 변화시키는 작용이 더 이상 보이지 않는다. 

- 람다 계산법에서의 프로그래밍 언어에서 람다식은 다음 $3$ 가지 방식에 불과하다. 

  1. $x$ (수)

  2. $\lambda x.E$ ($E$ 는 람다식, $x$ 는 수)

  3. $E_1 E_2$ (나란히 있는 람다식)

  - 이러한 람다식을 계산하는 방법은 $\lambda x.E y$ 에서 람다식 $E$ 에 포함된 모든 $x$ 를 $y$ 로 바꾸고 $\lambda x$ 를 없애는 것이다. 
  
  - $1$ 이나 $2$ 유형이 되면 더 이상 대입할 입력이 없으므로 그만 둔다. 오직 $3$ 유형에서만 대입이 일어날 수 있기 때문이다. 

    - 예시

      $$ (\lambda x.(\lambda y.xy))z \to  \lambda y.zy $$
  
  - 이처럼 간단한 람다 계산법은 튜링-완전하여 컴퓨터로 실행할 수 있는 모~든 것을 다 표현할 수 있다. 

- 이 순수 람다 계산법으로 자연수, 사칙연산, 참과 거짓, "그리고", "또는", "아닌" 을 표현할 수 있다.

# <a name="프로그래밍 언어와 논리" href="#프로그래밍 언어와 논리">프로그래밍 언어와 논리</a>

- 그런데 람다 계산법은 프로그래밍에 관한 놀라운 사실을 밝혀주었다. 

  - 그것은 프로그래밍 언어와 논리는 정확히 같은 것이다는 사실이다. 

  - 이는 프로그래밍 언어로 짠 프로그램이 논리적 증명과 일대일로 정확히 대응된다는 것이다. 

  - 지금까지 프로그래밍을 해왔다면 수학에서의 논리적 증명을 해온 것이고, 수학에서의 논리적 증명을 해왔다면 프로그래밍을 해온 것이다. 

  > 프로그램 $\Leftrightarrow$ 증명 $\dots$ 취약점 $\Leftrightarrow$ 허술한 증명 $\dots$ 기계적 계산을 정의한 프로그래밍 언어로부터 취약점 존재를 유일하게 결정할 수 있는 증명법이 있을까? 

- 유클리드의 원론과 러셀의 수학원리와 힐베르트 프로그램에서 알아보았듯, 수학에서의 논리적 증명이란 공리에 추론규칙을 적용하여 정리를 연역하는 과정이었다. 

  - 프로그램도 이 수학적 증명에 정확히 대응되어 다음과 같이 이처럼 논리적으로 연역하는 과정의 연속이다.

  |증명하기|프로그램짜기|
  |:---:|:---:|
  |공리로부터 정리를 연역하는 과정|기존의 데이터로부터 새로운 데이터를 만드는 과정|
  |참 또는 참이라고 가정한 것에서 시작함|기초적 데이터(프로그래밍 언어가 기본적으로 제공하는 정수, 문자, 참, 거짓 등)에서 시작함.|
  |사실을 기반으로 새로운 정리를 연역함|이미 있는 데이터로부터 새로운 데이터를 만들어감|

  ### 아 이것은 형식적 연역체계에서 형식적 추론 규칙으로 연역될 수 있는 증명법을 말하고 있구나. 그러므로 프로그래밍은 형식체계 $PM$ 에서 형식적 추론규칙으로 통해 무의미한 기호를 조작하는 과정에 불과하구나. 또한 그렇기에 불완전성 정리 밑에 있는거고. 즉, 모든 진리를 만들어낼 수 없다는 것, 문제를 풀 수 없다는 것.

## <a name="논리 추론 예시 " href="#논리 추론 예시 ">논리 추론 예시 </a>

- 프로그래밍이 어떻게 수학에서의 논리적 증명과 대응되는지 살펴보기 전에 대표적인 논리 추론 유형을 몇 가지 살펴보자. 

1. $A$ 가 사실이고 $B$ 가 사실이면 "$A$ 그리고 $B$" 도 사실이다. 

    $$ (A, B) \to A \land B $$

2. "$A$ 그리고 $B$" 가 사실이면 $A$ 도 사실이고 $B$ 도 사실이다. 

    $$ A \land B \to A $$

    $$ A \land B \to B $$

3. $A$ 가 사실이라고 가정하에 $B$ 가 사실이면 "$A$ 이면 $B$" 도 사실이다. 

    $$ \overline{A} \dots B \to A \Rightarrow B $$

    - 가정한 것에는 윗줄을 긋는다.

4. "$A$ 이면 $B$" 가 사실이고 $A$ 가 사실이면 $B$ 도 사실이다. 

    $$ (A \Rightarrow B, A) \to B $$

5. $A$ 가 사실이면 임의의 $B$ 에 대한 "$A$ 또는 $B$" 도 사실이고 "$B$ 또는 $A$" 도 사실이다. 

    $$ A \to A \lor B $$

    $$ A \to B \lor A $$

6. $A$ 또는 $B$ 가 사실이고 $A$ 가 사실이라는 가정하에 $C$ 가 사실이고 $B$ 가 사실이라는 가정하에 $C$ 가 사실이면 $C$ 는 사실이다. 

    $$ (A \lor B, \overline{A} \dots C, \overline{B} \dots C) \to C $$

- 이와 같은 논리 추론 규칙들로 무수히 많은 새로운 정리를 연역해낼 수 있다. 

  - 가령 두 개의 전제 

    $$ A, (A \Rightarrow B)\land (A \Rightarrow C) $$

    로부터 

    $$ (A \Rightarrow B) \land (A \Rightarrow C) \Rightarrow (A \Rightarrow (B \land C)) $$

    를 이끌어낼 수 있다.

| $4$ | $(A \Rightarrow B, A) \to B$ | 함수를 사용하는 것 |

- 유형 $3$ 

  - 인자($A$ 데이터) 가 주어지면 그것으로 새로운 데이터($B$ 데이터)를 만드는 함수이다. 

  - 예시 

    함수 `pair(x) = (x-1, x+1)` 는 정수 $x$($A$ 데이터) 로 튜플($B$ 데이터) 를 만들도록 프로그래밍 되었다. 

- 유형 $4$ 

  - $A$데이터를 주면 $B$데이터를 만드는 함수가 있을 때 이 함수에 $A$데이터를 주고 $B$데이터를 만드는 것이다.

  - 예시 
  
    함수 `pair` 에 정수 `3` 을 대입하여 `pair(3)`, 즉 `(2, 4)` 라는 데이터를 만든다. 

|유형|논리추론|프로그래밍| 
|:---:|:---:|:---:|
|$5$| $A \to (A \lor B, B \lor A)$ | 프로그램에서 데이터를 추상화시키는 것. |
|$6$| $(A \lor B, \overline{A} \dots C, \overline{B} \dots C) \to C$ | 프로그램에서 추상화된 데이터를 사용하는 것. |

- 유형 $5$ 

  - 예시 
  
    사과($A$데이터) 가 있을 때 사과를 그것이 배($B$ 데이터)와 함께 있을 때 추상적으로 표현되는 과일($A \lor B$)로 추상화 시킨다. 그래서 $A \lor B$ 속에는 $A$데이터 혹은 $B$데이터 하나만 존재한다. 

- 유형 $6$ 

  - 예시 
  
    $A \lor B$ 라고 추상화된 데이터가 있을 때 이것이 $A$데이터일 경우 $C$데이터를 만들고 $B$데이터일 경우 $C$데이터를 만드는 프로그램이 있다면, 이 프로그램으로 추상화된 데이터 $A \lor B$ 를 받아서 각 경우에 따라 $C$데이터를 만들 수 있다. 

- 이러한 방식으로 프로그래밍 언어가 제공하는 모든 것이 논리적 추론과 정확히 일대일 대응된다.

  - 코드 한 줄이 형식체계의 형식적 증명의 한 줄이었고, 증명의 한 줄은 프로그램 한 줄이었던 것이다. 

  - 이 대응관계를 하스켈 커리와 윌리엄 하워드가 밝형내서 커리-하워드 동형Curry-Howard isomorphism 또는 커리-하워드 대응Curry-Howard correspondence 이라고 한다. 

## <a name="커리-하워드 동치의 의미" href="#커리-하워드 동치의 의미">커리-하워드 동치의 의미</a>

- 이 대응관계가 튜링기계 패러다임이 아닌 람다 계산법 패러다임의 프로그래밍 언어에서 밝혀진 것은

  논리추론의 관점으로 프로그래밍 언어를 바라보았을 때 람다 패러다임의 프로그램들이 논리적 증명과 대응되는 모습을 더욱 잘 보여주었기 때문이다. 

  - 반면 튜링기계 패러다임의 프로그램들과 논리적 증명을 대응했을 때는 그 대응관계를 쉽게 알아볼 수 없었다.
  
  - 기계에 전달하는 일련의 명령어들은 논리적 증명과 거리가 멀어보이기 때문이다. 

- 이 대응관계를 다음과 같이 정리할 수 있다.

  |논리|프로그래밍| 
  |:---:|:---:|
  |증명|프로그램|
  |증명이란 어떤 형식문을 연역하는 과정이다.|프로그램은 어떤 데이터를 만드는 과정이다.|
  |증명에는 결론에 해당하는 참인 형식문이 있다.|프로그램에는 최종적으로 만드는 데이터의 종류가 있다.|
  |증명한 형식문(정리)|프로그램의 타입type|

- 이 대응관계는 프로그래머에게 실질적인 도움을 주었다.

  - 논리적 증명 과정에서 중요한 것은 올바른 식을 쓰고 올바르게 추론규칙을 적용하여 올바른 증명을 하는 것이다. 

  - 소프트웨어 제작도 이것과 똑같이 올바른 코드를 작성해야 올바르게 작동하고 실수가 없다.

  - 프로그래머는 프로그램을 짜기 전에 프로그램을 설계할 때 제대로 설계했는지 몰라서 불안하고, 실제로 프로그램을 짠 후에도 제대로 실행할지 몰라서 불안하다.

- 그런데 프로그램을 논리적 증명에 대응시킴으로써 이 불안함을 해소해줄 수 있다.

  - 이 기술의 개발은 아직도 현재진행형이다.
